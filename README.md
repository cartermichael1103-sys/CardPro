# Sports Card Break Value Index

A static website tracking Break Value Scores, trends, and buy/hold/sell
signals for top prospects and stars across MLB, NBA, NFL, and soccer.

The Google Sheet is the source of truth you edit by hand (or via your own
eBay Apps Script pulling sold listings). When you want the website to
reflect the latest Sheet data, you trigger a manual GitHub Action that
pulls the Sheet, regenerates `docs/data/players.json`, and commits it.
GitHub Pages then serves the updated site automatically.

## One-time setup

### 1. Create the Google Sheet
Create a tab named exactly `Players` with a header row (any order) using
these column names:

```
rank, player, sport, avg_sale, psa10_avg, sales_per_week, liquidity_score,
bvs, prospect_grade, rookie_year, rookie_card_year, bowman_1st,
chrome_available, prizm_available, optic_available, topps_available,
trend_30day, signal, risk, notes
```

Notes:
- `trend_30day` values should be one of: `up_strong`, `up`, `flat`, `down`
  (these map to the 📈📈 / 📈 / ➡️ / 📉 icons on the site)
- `signal` should be `Buy`, `Hold`, or `Sell`

### 2. Create a Google Service Account
1. Go to https://console.cloud.google.com/ and create (or pick) a project.
2. Enable the **Google Sheets API** for that project.
3. Go to IAM & Admin → Service Accounts → Create Service Account.
4. Once created, open it → Keys → Add Key → Create new key → JSON.
   This downloads a `.json` file — keep it private, never commit it.
5. Open the JSON file and copy the `client_email` value.
6. In your Google Sheet, click Share and paste that email in as a Viewer.

### 3. Add GitHub repo secrets
In your GitHub repo: Settings → Secrets and variables → Actions → New
repository secret. Add two secrets:

- `GOOGLE_SERVICE_ACCOUNT_JSON` — paste the **entire contents** of the
  service account JSON file
- `SHEET_ID` — the long ID from your Sheet's URL:
  `https://docs.google.com/spreadsheets/d/THIS_PART_HERE/edit`

### 4. Enable GitHub Pages
Settings → Pages → Source: Deploy from a branch → Branch: `main`,
folder: `/docs` → Save. Your site will be live at
`https://<your-username>.github.io/<repo-name>/` within a minute or two.

## Day-to-day use

1. Update your Google Sheet (manually or via your own scripts).
2. Go to your repo's **Actions** tab → **Update Player Data** → **Run workflow**.
3. Wait ~30 seconds. The site updates automatically once GitHub Pages
   picks up the new commit.

## Local structure

```
docs/                  ← served by GitHub Pages
  index.html
  style.css
  app.js
  data/players.json    ← regenerated by the workflow, don't hand-edit
scripts/
  pull_sheet_data.py   ← pulls Sheet → writes players.json
  requirements.txt
.github/workflows/
  update-data.yml      ← manual-only trigger (workflow_dispatch)
```

## Roadmap / upgrade path

- Sample data currently ships in `docs/data/players.json` so the site
  works before you connect a real Sheet — replace it by running the
  workflow once secrets are set up.
- When you outgrow a static site (want per-player pages, search-engine
  friendly URLs, server-side rendering), this same `docs/data/players.json`
  can become a build-time data source for Next.js or Astro without
  changing anything upstream (Sheet, service account, workflow).
- If you outgrow Sheets itself (10,000+ rows, relational queries across
  Players/Cards/Products), migrate to Supabase (free Postgres tier) —
  the workflow's "pull → JSON → commit" pattern stays the same, you'd
  just swap the Sheets API call for a Postgres query.
